# Version v0.1.x

OVDS v0.1.x defines the initial public structure for:

- reusable groups
- reusable makes
- reusable platforms
- concrete vehicle records
- normalized safety feature records
- reusable engines
- reusable electric motors
- reusable battery packs
- reusable transmissions

The vehicle powertrain model in this version is role-based and separates energy sources, storage, converters, and traction motors.

This version leaves fuel-cell stacks as a forward-compatible reference inside the vehicle powertrain model, without introducing a dedicated reusable schema yet.

This version also separates published performance results from compliance data, so efficiency, range, and emissions can be represented independently from emissions or safety certifications.

It allows both aggregate and source-specific test-cycle labels in published performance results, including `epa` plus EPA subcycles such as `ftp` and `hfet`.

It also supports richer emissions reporting for greenhouse gases, criteria pollutants, and evaporative hydrocarbons.

It also models fuel capacity within powertrain energy storage and supports a separate ten-year cost-of-ownership estimate for energy, maintenance, and total cost.

It includes published cargo capacity, ground clearance, door count, brake type, and track-width measurements within `specs`, plus acceleration, top speed, torque, and power results within `performance`.

It also allows reusable engine and electric-motor records to carry component-level power values in addition to published vehicle-level power results.

It also includes charging interfaces and charging-time scenarios within `configuration`, with separate AC and DC capability.

It also includes a dedicated `origin` block for published vehicle assembly origin, kept separate from commercial lineage and technical configuration.

## Release Notes v0.1.2

OVDS v0.1.2 is a backward-compatible patch release within the v0.1 schema line.

This release adds a dedicated technical block for published emissions-control hardware and diagnostics in the concrete vehicle configuration.

### v0.1.2 Highlights

- Adds `configuration.emissionsControls` for catalytic converters, oxygen sensors, EGR, secondary air injection, onboard diagnostics, durability standard, and particulate-filter fitment.
- Keeps emissions-control hardware separate from `performance.emissions`, which remains focused on reported pollutant results by cycle and scope.
- Adds a v0.1.2 MHEV example that demonstrates the new emissions-controls block.
- Extends efficiency units to support mass-based reporting for gaseous-fuel ICE examples such as CNG.
- Adds dedicated diesel ICE and CNG ICE reference examples to complement the electrified examples.

## Release Notes v0.1.0

OVDS v0.1.0 is the first tagged public release of the Open Vehicle Data Standard.

This release establishes the initial stable schema set, the normative semantics for the model, the repository documentation structure, and a reproducible validation workflow using uv.

### v0.1.0 Highlights

- Introduces a concrete vehicle record with separate lineage, configuration, origin, specs, performance, cost-of-ownership, safety, compliance, and attributes blocks.
- Defines reusable schemas for groups, makes, platforms, engines, electric motors, battery packs, transmissions, and normalized safety features.
- Uses a role-based powertrain model that separates energy sources, storage, converters, and traction motors.
- Separates published performance results from compliance classifications.
- Allows performance results to distinguish aggregate cycle labels such as `epa` from EPA subcycles such as `ftp` and `hfet`.
- Adds a dedicated origin block for published assembly country, region, and plant information.
- Includes cargo capacity, ground clearance, door count, brake type, and track-width measurements in `specs`, plus acceleration, top speed, torque, and power in `performance`.
- Allows reusable engine and electric-motor schemas to carry component-level power while keeping system power in the vehicle performance block.
- Adds charging interfaces, separate AC/DC charging capability, and charging-time scenarios to the vehicle configuration model.
- Documents human-readable, semantic OVDS identifiers and clarifies that one commercial variant may map to multiple concrete vehicle records.
- Includes valid EV and hybrid reference examples plus a local validation script and an intentionally invalid payload for negative testing.

### v0.1.0 Examples

The release includes example payloads under [examples/v0.1.1](../../examples/v0.1.1):

- [examples/v0.1.1/ev.json](../../examples/v0.1.1/ev.json): canonical fictional battery-electric reference vehicle
- [examples/v0.1.1/mhev.json](../../examples/v0.1.1/mhev.json): mild-hybrid example with combustion engine, battery, electric assist motor, and manual transmission
- [examples/v0.1.1/hev.json](../../examples/v0.1.1/hev.json): full-hybrid power-split example with combustion engine and traction motor
- [examples/v0.1.1/phev.json](../../examples/v0.1.1/phev.json): plug-in hybrid example with external charging plus combustion and electric propulsion
- [examples/v0.1.1/series-hybrid.json](../../examples/v0.1.1/series-hybrid.json): series-hybrid example where the combustion engine acts as a generator and the electric motor drives the wheels
- [examples/v0.1.1/invalid.json](../../examples/v0.1.1/invalid.json): intentionally invalid example for negative validation testing

Together, these examples show how the same vehicle model can express distinct electrification strategies while `vehicle.id` continues to identify a unique published configuration record.

### Deferred For Future Versions

The following areas are intentionally left for later versions:

- dedicated schemas for fuel-cell stacks
- richer safety taxonomy beyond grouped booleans
- explicit schemas for model, generation, and variant entities
- fitment or application records across multiple vehicles and markets
- richer market-specific availability and packaging semantics

## Release Notes v0.1.1

OVDS v0.1.1 is a backward-compatible patch release within the v0.1 schema line.

This release publishes the schema family under `spec/v0.1.1/` and includes additive improvements to the vehicle model, reusable technical entities, documentation, and reference examples.

### v0.1.1 Highlights

- Expands `specs` with cargo capacity, ground clearance, door count, front/rear brake type, and front/rear track-width measurements.
- Expands `performance` with acceleration, top speed, torque, and vehicle-level power results.
- Distinguishes aggregate EPA results from source-cycle labels such as `ftp` and `hfet`.

### Compatibility

The v0.1.1 release is additive and intended to remain compatible with existing v0.1 payloads.

- No schema family directory change is required.
- Existing `spec/v0.1.1/` identifiers remain valid for this release.
- New fields are optional unless already required by the existing v0.1 model rules.
